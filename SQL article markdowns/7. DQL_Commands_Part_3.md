# Day 7: DQL Part 3 — Joining Tables

*A beginner's guide to databases for aspiring data analysts and data scientists*

Welcome to Day 7 — the most transformative day in the series. Every query so far has drawn from a single table. The real analytical questions in `online_store` cut across multiple tables at once: *What is the revenue per category — by name, not by ID? Which customers spent the most? Which products have never been sold?* These questions live across the boundaries between tables, and **`JOIN`** is how you cross them.

Joins are where the "relational" in relational database finally pays off. All the work on Day 1 (primary keys, foreign keys, relationships) and Day 2 (the five-table schema) was building toward this moment. If you understand keys, you already understand the logic of joins — today is just learning the syntax for what you already know conceptually.

All queries are MySQL-compatible and run on the `online_store` database from Day 4.

---

## 1. What is a join?

A **join** combines rows from two or more tables into a single result, based on a matching condition — almost always the foreign-key relationship between them.

On their own, our tables give us IDs, not names. The `orders` table knows `customer_id = 1`, but not that customer 1 is Aarav Sharma. The `order_items` table knows `product_id = 4`, but not that it's Noise-Cancelling Headphones costing ₹7,999. A join brings those tables together so you see the complete picture in one result.

The matching condition is written in an `ON` clause and almost always uses an `=` between a foreign key in one table and the primary key it references in the other:

```sql
-- General shape
SELECT ...
FROM   table_a
JOIN   table_b ON table_a.foreign_key = table_b.primary_key;
```

There are five kinds of join, and they differ in *which rows survive when there is no match*:

| Join type | Rows returned |
|-----------|--------------|
| `INNER JOIN` | Only rows where the condition matches in **both** tables |
| `LEFT JOIN` | All rows from the **left** table; `NULL` where the right has no match |
| `RIGHT JOIN` | All rows from the **right** table; `NULL` where the left has no match |
| `CROSS JOIN` | Every possible combination of rows from both tables |
| `SELF JOIN` | A table joined to itself using two aliases |

MySQL does not support `FULL OUTER JOIN` directly — we'll cover the workaround later.

---

## 2. Table aliases — essential notation

When querying multiple tables, column names can clash (`both customers and orders have customer_id`) and long table names repeated throughout a query become tedious. **Table aliases** solve both problems by giving each table a short nickname for the duration of the query:

```sql
SELECT c.first_name, o.order_date
FROM   customers c
JOIN   orders    o ON c.customer_id = o.customer_id;
```

Here `c` is the alias for `customers` and `o` for `orders`. The dot notation `c.first_name` means "the `first_name` column from the table aliased `c`." You don't need `AS` for table aliases — just write the alias after the table name.

From this point on, every multi-table query uses aliases. Get comfortable with them quickly — a single-letter or two-letter alias per table is the standard style:

| Table | Alias used in this series |
|-------|--------------------------|
| `customers` | `c` |
| `categories` | `cat` |
| `products` | `p` |
| `orders` | `o` |
| `order_items` | `oi` |

---

## 3. INNER JOIN — matching rows from both tables

`INNER JOIN` (often written as just `JOIN`, the two are identical) returns only rows where the `ON` condition finds a match in **both** tables. Rows from either table that have no counterpart in the other are excluded.

**Show products with their category names.** On their own, `products` only stores `category_id`. Joining with `categories` replaces the number with a readable name:

```sql
SELECT
    p.product_id,
    p.product_name,
    cat.category_name,
    p.price
FROM products p
INNER JOIN categories cat ON p.category_id = cat.category_id
ORDER BY cat.category_name, p.price;
```

All 15 products return, each now labelled with their category name — "Books", "Clothing", "Electronics" — instead of 1, 2, 3. Every product has a valid `category_id`, so nothing is excluded.

**Show customers alongside their orders.** Joining `customers` to `orders`:

```sql
SELECT
    c.first_name,
    c.last_name,
    o.order_id,
    o.order_date,
    o.status
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id
ORDER BY c.last_name, o.order_date;
```

This returns **15 rows** — one per order. Notice that customer 10, Noah Brown, does not appear at all. He has placed no orders, so there is no matching row in `orders` for him — `INNER JOIN` excludes him entirely. This is the defining behaviour of `INNER JOIN`: if a row has no match on the other side, it vanishes from the result.

**Connect order items to the product they represent:**

```sql
SELECT
    oi.order_id,
    p.product_name,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price AS line_total
FROM order_items oi
INNER JOIN products p ON oi.product_id = p.product_id
ORDER BY oi.order_id;
```

Returns all 25 order-item rows, now with the product name instead of `product_id = 4`. The calculated column `line_total` works just as it did on a single table — `SELECT` can compute expressions from any column in any of the joined tables.

## 4. LEFT JOIN — all rows from the left table

`LEFT JOIN` keeps every row from the **left table** (the one in the `FROM` clause), regardless of whether a match exists in the right table. Where there is no match, all right-table columns return `NULL`.

This makes `LEFT JOIN` the join you reach for when you need to include rows even if they have no corresponding entry on the other side.

**All customers with their order count — including customers who ordered nothing:**

```sql
SELECT
    c.customer_id,
    c.first_name,
    c.last_name,
    COUNT(o.order_id) AS orders_placed
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.first_name, c.last_name
ORDER BY orders_placed DESC;
```

| customer_id | first_name | last_name | orders_placed |
|-------------|------------|-----------|---------------|
| 1 | Aarav | Sharma | 3 |
| 2 | Diya | Patel | 2 |
| 3 | Rohan | Mehta | 2 |
| 4 | Ananya | Reddy | 2 |
| 5 | Vivaan | Singh | 2 |
| 6 | Ishaan | Gupta | 1 |
| 7 | Saanvi | Iyer | 1 |
| 8 | Kabir | Khan | 1 |
| 9 | Mia | Williams | 1 |
| 10 | Noah | Brown | 0 |

All 10 customers appear. Noah is there, showing 0 orders. This is the crucial difference from `INNER JOIN`, which would have returned only 9 rows and left Noah completely invisible.

A key subtlety: `COUNT(o.order_id)` — not `COUNT(*)` — correctly returns 0 for Noah. When there is no matching order, all `o.*` columns are `NULL`. `COUNT(o.order_id)` skips `NULL` values, so it counts zero. `COUNT(*)` would count the row itself and return 1, which would be wrong. When using `COUNT` with a `LEFT JOIN`, always count a column from the right table, not `*`.

**The "find non-matching rows" pattern — LEFT JOIN's killer use case.**

One of the most useful analytical patterns in SQL: find rows in table A that have *no corresponding row* in table B. You do this by joining with `LEFT JOIN` and then filtering for `NULL` in the right table's column.

*Find customers who have never placed any order:*

```sql
SELECT
    c.customer_id,
    c.first_name,
    c.last_name
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;
```

| customer_id | first_name | last_name |
|-------------|------------|-----------|
| 10 | Noah | Brown |

Only Noah. The `WHERE o.order_id IS NULL` condition keeps only the rows where the right table had no match — exactly the "has no orders" case we want.

*Find products that have never been ordered:*

```sql
SELECT
    p.product_id,
    p.product_name,
    cat.category_name,
    p.price
FROM products p
LEFT JOIN order_items oi  ON p.product_id   = oi.product_id
INNER JOIN categories cat ON p.category_id  = cat.category_id
WHERE oi.order_item_id IS NULL;
```

| product_id | product_name | category_name | price |
|------------|--------------|---------------|-------|
| 13 | Non-Stick Frying Pan | Home & Kitchen | 1599.00 |

The Non-Stick Frying Pan is the only product that appears in no order lines. We can now see it by name, in its category, with its price — something impossible from `order_items` alone, since a never-ordered product leaves no trace there.

### The ON vs WHERE trap with LEFT JOIN

This is the most important subtlety of `LEFT JOIN`, and it causes real bugs in production code.

When you add a filter in a `WHERE` clause to a `LEFT JOIN`, it can accidentally turn the left join into an inner join by eliminating `NULL` rows. Consider:

```sql
-- WRONG: intends to show all customers, but only their completed orders
SELECT c.first_name, c.last_name, o.order_date
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.status = 'Completed';
```

This returns only 9 customers because `WHERE o.status = 'Completed'` runs after the join and excludes rows where `o.status` is `NULL` — exactly the rows for customers with no orders (like Noah) or customers whose orders aren't completed. You've accidentally lost the "left" behaviour.

The fix: put the condition in the `ON` clause instead of `WHERE`:

```sql
-- RIGHT: all customers shown, only completed orders joined alongside them
SELECT c.first_name, c.last_name, o.order_date
FROM customers c
LEFT JOIN orders o
    ON  c.customer_id = o.customer_id
    AND o.status = 'Completed';
```

Now `o.status = 'Completed'` is part of the join condition, not a post-join filter. Customers with no completed orders still appear — they just get `NULL` for `order_date`. The rule to remember: **filter conditions that should preserve unmatched rows belong in `ON`; conditions that should remove rows from the final result belong in `WHERE`.**

## 5. RIGHT JOIN — the mirror image

`RIGHT JOIN` is the exact mirror of `LEFT JOIN`: it keeps every row from the **right** table, regardless of whether a match exists in the left table.

```sql
SELECT
    c.first_name,
    c.last_name,
    o.order_id,
    o.order_date
FROM customers c
RIGHT JOIN orders o ON c.customer_id = o.customer_id;
```

In practice, `RIGHT JOIN` is rarely used. Any `RIGHT JOIN` can be rewritten as a `LEFT JOIN` by swapping the table order, which is always clearer to read. If you find yourself reaching for `RIGHT JOIN`, flip the tables and use `LEFT JOIN` instead.

## 6. Joining three or more tables

Joins chain. You can join as many tables as you need, and each new `JOIN ... ON` clause links one more table into the result. MySQL evaluates them left to right.

The grand join of `online_store` — all five tables in one query, showing the full purchase journey from customer name to product and category:

```sql
SELECT
    c.first_name                     AS customer,
    o.order_id,
    o.order_date,
    o.status,
    p.product_name                   AS product,
    cat.category_name                AS category,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price      AS line_total
FROM customers c
INNER JOIN orders      o   ON c.customer_id  = o.customer_id
INNER JOIN order_items oi  ON o.order_id     = oi.order_id
INNER JOIN products    p   ON oi.product_id  = p.product_id
INNER JOIN categories  cat ON p.category_id  = cat.category_id
ORDER BY o.order_date, o.order_id;
```

This returns all **25 rows** from `order_items`, now enriched with the customer's name, the order date and status, the product name, and its category — pulling from all five tables in one readable result. This is the query that turns a database into an analyst's workspace.

When reading a multi-join query, trace the chain of foreign keys: `customers` → (via `customer_id`) → `orders` → (via `order_id`) → `order_items` → (via `product_id`) → `products` → (via `category_id`) → `categories`. Each `ON` clause is one step along that chain.

## 7. JOIN with GROUP BY — the analytical powerhouse

The most useful pattern in analytical SQL is combining `JOIN` with `GROUP BY`. Joins bring the right columns together across tables; `GROUP BY` then summarises them. Together, they answer the richest questions.

**Revenue by category name.** On Day 6 we computed revenue grouped by `category_id`. Now we can show the actual names:

```sql
SELECT
    cat.category_name,
    COUNT(DISTINCT o.order_id)          AS total_orders,
    SUM(oi.quantity)                    AS units_sold,
    SUM(oi.quantity * oi.unit_price)    AS total_revenue
FROM categories cat
INNER JOIN products    p   ON cat.category_id = p.category_id
INNER JOIN order_items oi  ON p.product_id    = oi.product_id
INNER JOIN orders      o   ON oi.order_id     = o.order_id
GROUP BY cat.category_id, cat.category_name
ORDER BY total_revenue DESC;
```

| category_name | total_orders | units_sold | total_revenue |
|---------------|--------------|------------|---------------|
| Electronics | 8 | 12 | 25491.00 |
| Clothing | 6 | 12 | 9391.00 |
| Books | 8 | 11 | 6892.00 |
| Home & Kitchen | 4 | 10 | 3493.00 |
| Sports | 3 | 6 | 3196.00 |

Electronics accounts for over half the total revenue despite having the same number of orders as Books — driven by the high price of the Headphones. This is exactly the kind of insight a business would act on.

**Top customers by total spend.** The customer-level version, showing names and amounts:

```sql
SELECT
    c.first_name,
    c.last_name,
    COUNT(DISTINCT o.order_id)          AS orders_placed,
    SUM(oi.quantity * oi.unit_price)    AS total_spent
FROM customers c
INNER JOIN orders      o  ON c.customer_id = o.customer_id
INNER JOIN order_items oi ON o.order_id    = oi.order_id
GROUP BY c.customer_id, c.first_name, c.last_name
ORDER BY total_spent DESC;
```

| first_name | last_name | orders_placed | total_spent |
|------------|-----------|---------------|-------------|
| Aarav | Sharma | 3 | 15193.00 |
| Diya | Patel | 2 | 9498.00 |
| Rohan | Mehta | 2 | 6594.00 |
| Ananya | Reddy | 2 | 5094.00 |
| Kabir | Khan | 1 | 3598.00 |
| Vivaan | Singh | 2 | 3196.00 |
| Ishaan | Gupta | 1 | 2794.00 |
| Saanvi | Iyer | 1 | 1498.00 |
| Mia | Williams | 1 | 998.00 |

Noah Brown doesn't appear — he placed no orders, so there are no `order_items` rows to join to him. If you needed him in the list with a ₹0, replace the first `INNER JOIN` with a `LEFT JOIN`. The column totals sum to ₹48,463 — the same grand total we computed on Day 6, now attributed to individual customers by name.

## 8. SELF JOIN — a table joined to itself

A **self join** joins a table to a copy of itself. You give it two different aliases so MySQL can distinguish the two "copies." The classic use case is hierarchical data (employees and their managers in the same table), but it's useful any time you want to compare rows within the same table.

In our database, we can use it to find products that have at least one cheaper sibling in the same category — products that are not the lowest-priced in their group:

```sql
SELECT DISTINCT
    p1.product_name     AS product,
    p1.price,
    cat.category_name
FROM products p1
INNER JOIN products    p2  ON p1.category_id = p2.category_id
                          AND p1.price > p2.price
INNER JOIN categories  cat ON p1.category_id = cat.category_id
ORDER BY cat.category_name, p1.price;
```

`p1` and `p2` are two aliases for the same `products` table. The `ON` condition matches a product in `p1` to any product in `p2` that shares its category but has a lower price. The result is the 10 products that are not the cheapest in their own category — verified against each category's price range.

The key technique in any self join: add a condition on the joining column (here `AND p1.price > p2.price`) to avoid a product matching itself, and use `DISTINCT` if the same row could appear multiple times due to multiple cheaper siblings.

## 9. CROSS JOIN and FULL OUTER JOIN

**CROSS JOIN** produces every possible combination of rows from two tables with no `ON` condition. Five categories crossed with 15 products returns 75 rows. It is rarely used in analysis (though useful in some advanced scenarios like generating date ranges), so we won't explore it further here.

```sql
-- Every possible category-product pairing (75 rows)
SELECT cat.category_name, p.product_name
FROM categories cat
CROSS JOIN products p;
```

**FULL OUTER JOIN** returns all rows from *both* tables, using `NULL` where there is no match on either side. MySQL does not have a `FULL OUTER JOIN` keyword. The standard workaround is combining a `LEFT JOIN` and a `RIGHT JOIN` with `UNION`:

```sql
SELECT c.customer_id, c.first_name, o.order_id
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id

UNION

SELECT c.customer_id, c.first_name, o.order_id
FROM customers c
RIGHT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.customer_id IS NULL;
```

In our database, every order has a valid customer (because of the foreign key constraint), so a full outer join would return the same result as a left join. Full outer joins are most useful when both tables can have unmatched rows — in our design, the foreign key prevents that from happening on the orders side.

## 10. UNION and UNION ALL — stacking result sets

While joins combine tables *horizontally* (adding columns), **`UNION`** combines result sets *vertically* (adding rows). Both queries must return the same number of columns, in compatible data types.

A practical use: compare two time periods in a single readable table. Revenue in Q1 vs Q2 2024:

```sql
SELECT
    'Q1 2024'                           AS period,
    COUNT(DISTINCT o.order_id)          AS orders_placed,
    SUM(oi.quantity * oi.unit_price)    AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.order_date BETWEEN '2024-01-01' AND '2024-03-31'

UNION ALL

SELECT
    'Q2 2024',
    COUNT(DISTINCT o.order_id),
    SUM(oi.quantity * oi.unit_price)
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.order_date BETWEEN '2024-04-01' AND '2024-06-30';
```

| period | orders_placed | revenue |
|--------|---------------|---------|
| Q1 2024 | 8 | 26178.00 |
| Q2 2024 | 7 | 22285.00 |

**`UNION` vs `UNION ALL`:** `UNION` removes duplicate rows (running a deduplication step, which is expensive on large data); `UNION ALL` keeps all rows, including duplicates, and is significantly faster. In most analytical contexts — combining two different time periods, two different statuses, two separate reports — there are no duplicates to remove, and `UNION ALL` is the right choice. Use plain `UNION` only when you specifically need deduplication.

## 11. Putting it all together — the five-table capstone

Let's close with a query that answers a precise business question using every skill from Days 5, 6, and 7:

> *"Which customers bought Electronics products in completed orders? Show their name, how many Electronics orders they placed, how many units they bought, and their total Electronics spend — ranked by spend."*

```sql
SELECT
    c.first_name,
    c.last_name,
    COUNT(DISTINCT o.order_id)          AS electronics_orders,
    SUM(oi.quantity)                    AS units_bought,
    SUM(oi.quantity * oi.unit_price)    AS electronics_spend
FROM customers c
INNER JOIN orders      o   ON c.customer_id  = o.customer_id
INNER JOIN order_items oi  ON o.order_id     = oi.order_id
INNER JOIN products    p   ON oi.product_id  = p.product_id
INNER JOIN categories  cat ON p.category_id  = cat.category_id
WHERE o.status          = 'Completed'
  AND cat.category_name = 'Electronics'
GROUP BY c.customer_id, c.first_name, c.last_name
ORDER BY electronics_spend DESC;
```

| first_name | last_name | electronics_orders | units_bought | electronics_spend |
|------------|-----------|--------------------|--------------|--------------------|
| Aarav | Sharma | 3 | 5 | 13395.00 |
| Diya | Patel | 1 | 1 | 7999.00 |
| Rohan | Mehta | 1 | 2 | 3298.00 |
| Ananya | Reddy | 1 | 1 | 799.00 |

The total Electronics spend across completed orders is ₹25,491 — which matches the category-revenue number we computed earlier, confirming the data is consistent.

Let's trace the full execution for this query:

1. **FROM / JOINs** — build the wide result by chaining all five tables together: 10 customers × 15 orders × 25 order-item lines × 15 products × 5 categories.
2. **WHERE** — filter down to rows where the order is Completed and the category is Electronics. This leaves only order-item lines for Electronics products in completed orders.
3. **GROUP BY** — collapse to one row per customer.
4. **SELECT** — compute the three aggregates for each customer group.
5. **ORDER BY** — rank by electronics spend.

Notice that customers who ordered Electronics in non-completed orders (like Kabir Khan, whose Pending order had Clothing) don't appear. Customers who ordered only non-Electronics products in completed orders (like Saanvi Iyer and Mia Williams) don't appear either. Only the four who have at least one Electronics line item in a completed order make the cut.

### Quick recap of Day 7

- A **join** combines rows from two or more tables based on a matching condition, almost always a primary-key / foreign-key pair.
- Use **table aliases** (`customers c`, `orders o`) and **dot notation** (`c.first_name`) to avoid ambiguity in multi-table queries.
- **`INNER JOIN`** returns only rows with a match in both tables — unmatched rows are excluded.
- **`LEFT JOIN`** returns all rows from the left table; unmatched right-table columns become `NULL`. Use `COUNT(right_table_column)` — not `COUNT(*)` — to count correctly across a left join.
- The **`LEFT JOIN` + `IS NULL`** pattern finds rows in one table that have no corresponding row in another ("never ordered", "never sold").
- **Filter placement matters with `LEFT JOIN`**: conditions that preserve unmatched rows go in `ON`; conditions that remove rows go in `WHERE`.
- **`RIGHT JOIN`** is the mirror of `LEFT JOIN` — rewrite as `LEFT JOIN` by swapping table order.
- Chain `JOIN` clauses to query three or more tables; each `ON` is one step along the foreign-key chain.
- **`JOIN` + `GROUP BY`** is the most powerful analytical pattern in SQL — cross-table joins bring the right columns together, aggregation summarises them.
- A **self join** uses two aliases on the same table to compare rows within it.
- **`UNION ALL`** stacks result sets vertically (rows); prefer it over `UNION` unless deduplication is specifically needed.
