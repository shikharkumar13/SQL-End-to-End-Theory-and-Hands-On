# Day 4: DML — Bringing the Database to Life

*A beginner's guide to databases for aspiring data analysts and data scientists*

Welcome to Day 4. Yesterday you built the skeleton of `online_store` — five connected tables, standing empty, guarded by constraints. Today we use **DML, the Data Manipulation Language**, to fill that skeleton with data and learn how to change it. By the end you'll have a fully populated database, and that data is exactly what we'll query for the rest of the series.

Every query here is written for **MySQL**. If you ran yesterday's setup script, you can run today's straight after it.

Let's bring it to life.

---

## 1. What DML does

**DML (Data Manipulation Language)** is the family of commands that work with the *data inside* your tables — the contents, not the container. There are three:

| Command | What it does |
|---------|--------------|
| `INSERT` | Adds new rows to a table |
| `UPDATE` | Modifies existing rows |
| `DELETE` | Removes rows from a table |

One warning to keep in mind from the very start: **`UPDATE` and `DELETE` act on every row you don't filter out.** A `DELETE` without a `WHERE` clause empties the entire table. We'll come back to this — but burn it into memory now, because it's the single most common beginner accident.

---

## 2. INSERT: adding data

The `INSERT` command adds rows. Its standard form names the columns you're filling, then provides the values:

```sql
INSERT INTO table_name (column1, column2, ...)
VALUES (value1, value2, ...);
```

Let's populate the simplest table, **`categories`**. Notice we *don't* supply `category_id` — that column is `AUTO_INCREMENT`, so MySQL assigns 1, 2, 3, … automatically:

```sql
INSERT INTO categories (category_name)
VALUES ('Electronics');
```

That inserts one row. But you can insert **many rows at once** by separating each set of values with a comma — far more efficient than writing one statement per row:

```sql
INSERT INTO categories (category_name) VALUES
('Electronics'),
('Books'),
('Clothing'),
('Home & Kitchen'),
('Sports');
```

After this, our five categories have IDs 1 through 5, assigned in order.

A few essentials this example already shows:

- **Text values go in single quotes** (`'Books'`), numbers do not.
- **Always list your columns** — `INSERT INTO categories (category_name)`. You *can* omit the column list, but then you must supply every column in exact table order, which is fragile. Naming columns is clearer and safer; make it a habit.
- **Skip `AUTO_INCREMENT` columns** — let MySQL fill them.

## 6. UPDATE: changing existing data

`UPDATE` modifies rows that already exist. Its form is:

```sql
UPDATE table_name
SET column1 = value1, column2 = value2, ...
WHERE condition;
```

The `WHERE` clause decides *which* rows change. Suppose order 5 has shipped and we want to mark it completed:

```sql
UPDATE orders
SET status = 'Completed'
WHERE order_id = 5;
```

Only the row where `order_id = 5` is affected. You can change **several columns at once** by separating them with commas — say a product's price rises and stock is restocked:

```sql
UPDATE products
SET price = 6999.00, stock_quantity = 60
WHERE product_id = 4;
```

You can also update *many rows* with one statement. To give every product in the Books category (category 2) a price increase, you'd target them all with the `WHERE`:

```sql
UPDATE products
SET price = price * 1.10
WHERE category_id = 2;
```

Here `price = price * 1.10` raises each matching price by 10% — SQL can calculate the new value from the old one.

> **⚠️ The most important rule of `UPDATE`.** If you forget the `WHERE` clause, *every row in the table changes*. This:
> ```sql
> UPDATE products SET price = 0;
> ```
> would set **every** product's price to zero. Always write your `WHERE` first, and consider running a `SELECT` with the same condition beforehand to preview exactly which rows you're about to touch. MySQL Workbench even has a "safe update mode" that blocks `UPDATE`/`DELETE` without a key-based `WHERE` — a helpful guardrail for beginners.

*(If you ran the two example `UPDATE`s above and want to follow the rest of the series with the original numbers, simply re-run the relevant inserts or reset those values back.)*

## 7. DELETE: removing data

`DELETE` removes whole rows. Its form is short — and that's exactly why it's dangerous:

```sql
DELETE FROM table_name
WHERE condition;
```

To cancel and remove a specific order item — say the line with `order_item_id = 25`:

```sql
DELETE FROM order_items
WHERE order_item_id = 25;
```

The same life-or-death rule applies: **`DELETE FROM order_items;` with no `WHERE` deletes every row.** Always filter.

### Foreign keys protect you here too

Try to delete a customer who has placed orders, and MySQL will *stop you*:

```sql
DELETE FROM customers
WHERE customer_id = 1;
```

This fails, because orders reference customer 1 through a foreign key. Deleting the customer would leave those orders pointing at someone who no longer exists — an "orphaned" record. This is **referential integrity** doing its job: the database refuses to let your relationships break. (To remove such a customer you'd first delete their dependent rows, or design the foreign key with `ON DELETE CASCADE` so children are removed automatically — a choice to make carefully.)

> **`DELETE` vs `TRUNCATE` (from Day 3).** `DELETE` removes *chosen* rows via `WHERE` and can be undone inside a transaction. `TRUNCATE` instantly wipes *all* rows and resets `AUTO_INCREMENT`, but can't target specific rows. Use `DELETE` for surgical removal, `TRUNCATE` to empty a whole table fast.

---

## 8. A safety net: transactions

Because `UPDATE` and `DELETE` can do irreversible damage, MySQL (using the InnoDB engine) lets you wrap changes in a **transaction** — a group of changes you can review before saving, or undo entirely. This is the **TCL** family we previewed on Day 2.

```sql
START TRANSACTION;

UPDATE products SET price = price * 1.10 WHERE category_id = 2;

-- Inspect the result. Happy with it?
COMMIT;     -- permanently save the changes

-- Changed your mind instead?
ROLLBACK;   -- undo everything since START TRANSACTION
```

`COMMIT` makes the changes permanent; `ROLLBACK` discards them as if they never happened. For risky bulk changes, this is your seatbelt — start a transaction, run the change, check the data, and only then commit. We won't lean on transactions heavily in this analysis-focused series, but knowing they exist will one day save you from a painful mistake.

### Quick recap of Day 4

- **DML** manipulates the data inside tables: `INSERT` (add), `UPDATE` (change), `DELETE` (remove).
- `INSERT` adds one or many rows; name your columns, quote text and dates, and skip `AUTO_INCREMENT` IDs. Use `NULL` for missing optional values.
- Insert in **dependency order** so foreign keys resolve: categories and customers, then products, orders, and finally order_items.
- `UPDATE ... SET ... WHERE` changes rows; `DELETE FROM ... WHERE` removes them. **A missing `WHERE` affects every row** — always filter.
- **Foreign keys** block deletes that would orphan related rows, protecting referential integrity.
- **Transactions** (`START TRANSACTION` / `COMMIT` / `ROLLBACK`) let you undo risky changes.
