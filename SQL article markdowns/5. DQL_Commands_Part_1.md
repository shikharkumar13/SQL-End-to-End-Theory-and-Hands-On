# Day 5: DQL Part 1 — SELECT, Filtering, and Sorting

*A beginner's guide to databases for aspiring data analysts and data scientists*

Welcome to Day 5 — the day the series shifts into analysis gear. You've built the database (Day 3) and filled it with data (Day 4). Everything from here on is about asking that data real questions. Today we cover **DQL, the Data Query Language**, and by the end you'll be able to retrieve any slice of data you want: choosing which columns to see, filtering rows down to exactly the ones that matter, and presenting results in any order. These are the tools you'll use in literally every query you ever write.

Every query here runs on the `online_store` database exactly as we populated it on Day 4. All syntax is MySQL-compatible.

Let's start asking questions.

---

## 1. What is DQL — and what does SELECT do?

**DQL (Data Query Language)** is the family of SQL commands that read data. It consists of essentially one command: **`SELECT`**. Unlike DDL and DML, which create, modify, or remove things, `SELECT` never changes anything. It is purely about retrieval — asking the database a question and getting an answer back. This makes it safe to experiment with: running a `SELECT` on a production database carries no risk to the data.

`SELECT` is also the command you will use most as an analyst — far more than any other. Mastering it is the core skill of SQL.

Here is the full shape of a `SELECT` statement with everything it can contain, ordered as MySQL expects:

```sql
SELECT   column(s)
FROM     table_name
WHERE    condition(s)
ORDER BY column(s)
LIMIT    number;
```

A crucial thing to understand: even though you write `SELECT` first, the database does **not** execute these clauses in writing order. The internal execution order is:

```
FROM → WHERE → SELECT → ORDER BY → LIMIT
```

That is: MySQL first identifies the source table (`FROM`), then filters rows (`WHERE`), then picks the columns you asked for (`SELECT`), then sorts (`ORDER BY`), then cuts the list down (`LIMIT`). Knowing this order explains some behaviours that otherwise seem odd — for instance, you can't use a column alias you defined in `SELECT` inside your `WHERE`, because `WHERE` runs before `SELECT` has been processed. We'll point this out where it matters.

Today's article covers all of the above except `FROM` with joins, which gets its full day on Day 7.

## 2. Your first SELECT — choosing columns

The most basic `SELECT` picks all columns from a table using the asterisk `*`:

```sql
SELECT * FROM products;
```

The `*` means "every column." This returns all 15 rows and all 5 columns of the `products` table. It's useful for quick exploration, but in real analysis you almost never want every column — so the better habit is naming exactly the columns you need:

```sql
SELECT product_name, price, stock_quantity
FROM products;
```

This returns only those three columns across all 15 rows. Selecting specific columns keeps results focused, makes queries easier to read, and can be significantly faster on large databases where tables might have dozens of columns.

You can list columns in any order you like — the output follows your order, not the table's original order:

```sql
SELECT price, product_name
FROM products;
```

This returns the price column first, even though `product_name` comes first in the table definition. `SELECT` controls the shape of the output, not the underlying storage.

## 3. Column aliases with AS

A **column alias** gives a column a different label in the output. You define it with the `AS` keyword after the column name. Aliases are pure display — they rename the result column without touching the actual table:

```sql
SELECT
    product_name   AS 'Product',
    price          AS 'Price (₹)',
    stock_quantity AS 'In Stock'
FROM products;
```

The output headers will now read "Product", "Price (₹)", and "In Stock" rather than the raw column names. This is particularly useful when sharing results with non-technical stakeholders, or when you build calculated columns (more on those in later days). Aliases with spaces or special characters need to be wrapped in single quotes or backticks.

A small MySQL note: `AS` is optional — `price 'Price (₹)'` works the same as `price AS 'Price (₹)'`. But always writing `AS` makes your intent explicit and your code easier to read.

## 4. DISTINCT — removing duplicate values

When you select a column that has repeating values, the default is to show every row including repeats. `DISTINCT` keeps only one copy of each unique value:

```sql
SELECT DISTINCT country
FROM customers;
```

Our 10 customers span four countries. Without `DISTINCT`, country would appear once per customer (10 rows). With `DISTINCT`, you get exactly four rows: India, Pakistan, UK, USA.

```sql
SELECT DISTINCT status
FROM orders;
```

This returns the three status values present in our orders — Completed, Pending, Cancelled — regardless of how many orders have each status. Useful for a quick survey of what values a column actually contains.

`DISTINCT` applies to the combination of all selected columns, not just the first one:

```sql
SELECT DISTINCT country, city
FROM customers;
```

This returns each unique country-and-city pair. Because no two customers share both the same country and the same city, this returns all 10 rows (except Noah, whose city is `NULL` — which `DISTINCT` treats as one distinct value).

## 5. WHERE — filtering rows

The `WHERE` clause filters which rows appear in the result. Without it, a query returns every row in the table. With it, you get only the rows that satisfy your condition. This is where the real analytical power begins.

### Comparison operators

The most fundamental filters compare a column's value to something. MySQL supports all the standard operators:

| Operator | Meaning |
|----------|---------|
| `=` | Equal to |
| `!=` or `<>` | Not equal to |
| `>` | Greater than |
| `<` | Less than |
| `>=` | Greater than or equal to |
| `<=` | Less than or equal to |

Let's put them to work on our data.

Find all customers from India:

```sql
SELECT first_name, last_name, city, country
FROM customers
WHERE country = 'India';
```

This returns 7 rows — all customers whose `country` is exactly `'India'`. Notice that text values go in single quotes; comparisons are not case-sensitive by default in MySQL's standard character set.

Find products that cost more than ₹1,000:

```sql
SELECT product_name, price
FROM products
WHERE price > 1000;
```

This returns 7 products: the Mechanical Keyboard (₹2,499), USB-C Hub (₹1,299), Noise-Cancelling Headphones (₹7,999), Data Science Handbook (₹1,199), Denim Jeans (₹1,799), Hooded Sweatshirt (₹1,499), and Non-Stick Frying Pan (₹1,599).

Find orders placed on or after 1st April 2024:

```sql
SELECT order_id, customer_id, order_date, status
FROM orders
WHERE order_date >= '2024-04-01';
```

Dates in `WHERE` conditions follow the `'YYYY-MM-DD'` format, in single quotes. This returns orders 9 through 15 — the 7 orders placed in April, May, and June.

Find any product that is currently out of stock:

```sql
SELECT product_name, stock_quantity
FROM products
WHERE stock_quantity = 0;
```

Only one product matches: the Resistance Bands Set.

Find orders that are not completed:

```sql
SELECT order_id, customer_id, order_date, status
FROM orders
WHERE status != 'Completed';
```

This returns 4 rows — the 2 Pending and 2 Cancelled orders.

### AND, OR, NOT

Real filtering questions usually combine more than one condition. The logical operators `AND`, `OR`, and `NOT` let you build compound conditions.

**`AND`** requires both conditions to be true. Find affordable Electronics — products in category 1 (Electronics) priced under ₹2,000:

```sql
SELECT product_name, price
FROM products
WHERE category_id = 1 AND price < 2000;
```

This returns two products: the Wireless Mouse (₹799) and USB-C Hub (₹1,299). The Mechanical Keyboard and Headphones are Electronics but exceed ₹2,000, so they're filtered out.

**`OR`** requires at least one condition to be true. Find customers from either Pakistan or the UK:

```sql
SELECT first_name, last_name, country
FROM customers
WHERE country = 'Pakistan' OR country = 'UK';
```

This returns Kabir Khan (Pakistan) and Mia Williams (UK) — 2 rows.

**`NOT`** negates a condition. Find all products that are not in the Books category (category 2):

```sql
SELECT product_name, category_id
FROM products
WHERE NOT category_id = 2;
```

Returns the 12 products outside Books. `NOT` tends to read most naturally with `IN` and `LIKE`, which we'll see shortly.

**Combining AND, OR, and NOT — mind the parentheses.** Operator precedence in SQL is `NOT` first, then `AND`, then `OR` — same as algebra (and like multiplication before addition). When you mix them, use parentheses to make your intent explicit and unambiguous.

Compare these two queries:

```sql
-- Query A: WITHOUT parentheses
SELECT product_name, category_id, price
FROM products
WHERE category_id = 1 OR category_id = 2 AND price < 1000;

-- Query B: WITH parentheses — what you probably meant
SELECT product_name, category_id, price
FROM products
WHERE (category_id = 1 OR category_id = 2) AND price < 1000;
```

Query A (no parentheses) is evaluated as `category_id = 1 OR (category_id = 2 AND price < 1000)` — returns all Electronics regardless of price, plus only cheap Books. Query B returns only products that are in Electronics or Books AND cost under ₹1,000. They return different results. When in doubt, add parentheses — they cost nothing and prevent subtle errors.

### BETWEEN

`BETWEEN` tests whether a value falls within a range. Both endpoints are **inclusive**:

```sql
SELECT product_name, price
FROM products
WHERE price BETWEEN 500 AND 1500;
```

This returns 9 products priced from ₹500 to ₹1,500 (including exactly ₹500 and ₹1,500). `BETWEEN 500 AND 1500` is exactly equivalent to `price >= 500 AND price <= 1500` — just cleaner to read.

`BETWEEN` works on dates too, which makes it perfect for filtering time periods:

```sql
SELECT order_id, customer_id, order_date, status
FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31';
```

This returns the 7 orders placed in Q1 2024 (January through March). Date ranges in analysis are so common that `BETWEEN` on date columns becomes second nature.

To negate it, use `NOT BETWEEN`:

```sql
SELECT product_name, price
FROM products
WHERE price NOT BETWEEN 500 AND 1500;
```

Returns the 6 products outside that price band — Noise-Cancelling Headphones (₹7,999) at the top, Ceramic Coffee Mug (₹349) at the bottom.

### IN

`IN` tests whether a value matches any item in a list. It's the clean alternative to a long chain of `OR` conditions:

```sql
-- Without IN (verbose)
WHERE country = 'India' OR country = 'UK' OR country = 'USA'

-- With IN (clean)
SELECT first_name, last_name, country
FROM customers
WHERE country IN ('India', 'UK', 'USA');
```

Both are equivalent, but `IN` is far more readable as the list grows. This returns 9 of our 10 customers — everyone except Kabir Khan from Pakistan.

Find all orders that are pending or cancelled:

```sql
SELECT order_id, customer_id, order_date, status
FROM orders
WHERE status IN ('Pending', 'Cancelled');
```

Returns 4 orders. `NOT IN` reverses it — find everything that doesn't match the list:

```sql
SELECT order_id, customer_id, order_date, status
FROM orders
WHERE status NOT IN ('Pending', 'Cancelled');
```

Returns the 11 completed orders.

> **`NOT IN` and NULL — a trap to know.** If the list inside `NOT IN` contains a `NULL` value, the query returns no rows at all. This is because MySQL can't determine whether a value equals `NULL` (since `NULL` means unknown). For now, keep `NULL` values out of `IN` lists; we'll handle `NULL` properly with `IS NULL` next.

### LIKE — pattern matching

`LIKE` matches text patterns using two wildcard characters:

| Wildcard | Matches |
|----------|---------|
| `%` | Any sequence of zero or more characters |
| `_` | Exactly one character |

Find all customers whose first name starts with the letter A:

```sql
SELECT first_name, last_name, email
FROM customers
WHERE first_name LIKE 'A%';
```

Returns Aarav Sharma and Ananya Reddy — the two customers whose first name begins with A. The `%` matches anything that follows (including nothing).

Find all products with "Keyboard" anywhere in the name:

```sql
SELECT product_name, price
FROM products
WHERE product_name LIKE '%Keyboard%';
```

Returns the Mechanical Keyboard. The `%` on both sides means "Keyboard can appear anywhere in the name."

Find customers whose email ends with `.com`:

```sql
SELECT first_name, last_name, email
FROM customers
WHERE email LIKE '%.com';
```

Returns all 10 customers — every email in our dataset ends with `.com`.

The underscore `_` matches exactly one character. To find products whose name starts with any two characters followed by "B":

```sql
SELECT product_name
FROM products
WHERE product_name LIKE '__B%';
```

Use `NOT LIKE` to find rows that don't match the pattern:

```sql
SELECT first_name, last_name, email
FROM customers
WHERE email NOT LIKE '%gmail%';
```

Returns customers whose email doesn't contain "gmail" — in our data, all of them, since we used generic `@email.com` domains.

> **MySQL `LIKE` is case-insensitive by default** for the standard `utf8mb4_0900_ai_ci` character set, which is MySQL 8's default. So `LIKE 'a%'` and `LIKE 'A%'` give identical results. This is usually what you want for text searches.

### IS NULL and IS NOT NULL

Remember Noah Brown from Day 4, whose `city` was deliberately left as `NULL`? Here's where that pays off.

`NULL` means "no value" — and it has a special rule: **you cannot compare `NULL` using `=` or `!=`.** This is the single most common `NULL` mistake beginners make.

```sql
-- This will NEVER return any rows, even for NULL cities
SELECT * FROM customers WHERE city = NULL;  -- Wrong

-- This is the correct way
SELECT * FROM customers WHERE city IS NULL;  -- Right
```

`= NULL` always evaluates to "unknown" in SQL, so it never matches anything. `IS NULL` is the dedicated operator for detecting missing values:

```sql
SELECT first_name, last_name, country
FROM customers
WHERE city IS NULL;
```

Returns exactly one row: Noah Brown. His city was stored as `NULL` on Day 4.

The inverse finds all customers who do have a city on record:

```sql
SELECT first_name, last_name, city
FROM customers
WHERE city IS NOT NULL;
```

Returns 9 customers — everyone except Noah. Handling `NULL` cleanly is a mark of a careful analyst, because `NULL` can silently distort counts, averages, and joins if you don't account for it.

## 6. ORDER BY — sorting results

`ORDER BY` controls how the result rows are sorted. Without it, MySQL makes no guarantee about the order of rows — you might see them in insert order, or in any other order, and this can change between runs. If order matters to you, always state it explicitly.

Sort products from most expensive to least:

```sql
SELECT product_name, price
FROM products
ORDER BY price DESC;
```

`DESC` means descending (largest first). The top result is Noise-Cancelling Headphones at ₹7,999. The default direction is `ASC` (ascending, smallest first) — it's good practice to write it explicitly anyway for clarity.

Sort customers by their signup date, oldest first:

```sql
SELECT first_name, last_name, signup_date
FROM customers
ORDER BY signup_date ASC;
```

Returns customers in chronological order: Aarav Sharma (2023-01-15) at the top, Noah Brown (2024-02-14) at the bottom.

**Sorting by multiple columns** breaks ties. Sort customers by country alphabetically, then by city within each country:

```sql
SELECT first_name, last_name, city, country
FROM customers
ORDER BY country ASC, city ASC;
```

India has seven customers — they appear together, then sorted by city among themselves (Bengaluru, Chennai, Delhi, Hyderabad, Jaipur, Mumbai, Pune). Then Pakistan (Lahore), then UK (London), then USA. Noah's `NULL` city appears last within the USA group — MySQL sorts `NULL` values last in `ASC` order, and first in `DESC`.

**Sorting by a column you didn't select.** You can sort by any column in the table, not just the ones you listed in `SELECT`:

```sql
SELECT product_name, price
FROM products
ORDER BY stock_quantity ASC;
```

The output shows only `product_name` and `price`, but rows are sorted by `stock_quantity` even though that column isn't in the result — perfectly valid MySQL.

## 7. LIMIT — capping the number of rows

`LIMIT` restricts how many rows the query returns. It's essential for previewing data, building "top N" lists, and avoiding accidentally pulling millions of rows.

Get the 5 most expensive products:

```sql
SELECT product_name, price
FROM products
ORDER BY price DESC
LIMIT 5;
```

Returns the five priciest items: Noise-Cancelling Headphones (₹7,999), Mechanical Keyboard (₹2,499), Denim Jeans (₹1,799), Non-Stick Frying Pan (₹1,599), Hooded Sweatshirt (₹1,499). The `LIMIT` is applied after `ORDER BY`, so you always get the top 5 sorted correctly.

> Without `ORDER BY`, `LIMIT 5` returns an arbitrary 5 rows — which is rarely what you want. Always pair `LIMIT` with `ORDER BY` for predictable results.

**`LIMIT` with `OFFSET`** lets you skip a number of rows before starting the count. This is how you page through results:

```sql
-- First page (rows 1-5)
SELECT product_name, price
FROM products
ORDER BY price DESC
LIMIT 5 OFFSET 0;

-- Second page (rows 6-10)
SELECT product_name, price
FROM products
ORDER BY price DESC
LIMIT 5 OFFSET 5;
```

`OFFSET 5` skips the first 5 rows and returns the next 5. Useful for pagination in applications. MySQL also accepts a shorthand: `LIMIT 5, 5` means "skip 5, return 5" — though the explicit `LIMIT n OFFSET m` form is easier to read.

## 8. Putting it all together — a real analyst query

Now let's combine what we've learned into one query that answers a realistic business question:

> *"Which active-inventory products in Electronics or Books cost between ₹500 and ₹3,000 and have the name containing 'a'? Show the most expensive first."*

```sql
SELECT
    product_name    AS 'Product',
    price           AS 'Price (₹)',
    stock_quantity  AS 'Stock'
FROM products
WHERE
    category_id IN (1, 2)
    AND price BETWEEN 500 AND 3000
    AND stock_quantity > 0
    AND product_name LIKE '%a%'
ORDER BY price DESC
LIMIT 10;
```

Let's trace through the execution order to see what MySQL does step by step:

1. **FROM** — start with the full `products` table (15 rows).
2. **WHERE** — apply all four conditions together: category is Electronics or Books → 7 products remain; price between ₹500 and ₹3,000 → Headphones (₹7,999) filtered out → 6 remain; stock > 0 → Resistance Bands Set filtered out, but it's in Sports so already gone → 6 remain; name contains 'a' → removes SQL Fundamentals, The Pragmatic Programmer — leaves Wireless Mouse, Mechanical Keyboard, Data Science Handbook.
3. **SELECT** — return only the three aliased columns for those rows.
4. **ORDER BY** — sort by price descending: Mechanical Keyboard first, then Data Science Handbook, then Wireless Mouse.
5. **LIMIT** — return at most 10; we have 3, so all 3 return.

The result:

| Product | Price (₹) | Stock |
|---------|-----------|-------|
| Mechanical Keyboard | 2499.00 | 80 |
| Data Science Handbook | 1199.00 | 120 |
| Wireless Mouse | 799.00 | 150 |

This is how a real analyst works: start with the full table, filter down to exactly the rows and columns that matter, sort meaningfully, and cap the output to a useful size.

## 9. Quick reference — the WHERE operators

Here's everything from today in one lookup table for when you need it:

| What you need | Syntax | Example |
|---------------|--------|---------|
| Exact match | `= value` | `WHERE status = 'Completed'` |
| Not equal | `!= value` | `WHERE status != 'Cancelled'` |
| Comparison | `>`, `<`, `>=`, `<=` | `WHERE price >= 1000` |
| Both conditions | `AND` | `WHERE cat = 1 AND price < 2000` |
| Either condition | `OR` | `WHERE cat = 1 OR cat = 2` |
| Negate | `NOT` | `WHERE NOT cat = 2` |
| Range (inclusive) | `BETWEEN a AND b` | `WHERE price BETWEEN 500 AND 1500` |
| List match | `IN (a, b, c)` | `WHERE country IN ('India', 'UK')` |
| Pattern match | `LIKE 'pattern'` | `WHERE name LIKE 'A%'` |
| Missing value | `IS NULL` | `WHERE city IS NULL` |
| Present value | `IS NOT NULL` | `WHERE city IS NOT NULL` |

## Where we're headed

You now have full control over *which rows and columns* a query returns. But analysis almost always needs more than raw rows — it needs summaries. "How many orders per customer?" "What's the total revenue?" "Which category sells the most?" These questions require computing across groups of rows, not just reading individual ones.

That's **Day 6: DQL Part 2 — Aggregation and Grouping.** You'll meet `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`, and the `GROUP BY` and `HAVING` clauses that turn raw rows into meaningful summaries — the foundation of almost every analytical report.

---

### Quick recap of Day 5

- **DQL** reads data without changing it. `SELECT` is its only command, and it's the most-used command in all of SQL.
- Execution order is `FROM → WHERE → SELECT → ORDER BY → LIMIT` — not the writing order.
- **`SELECT column(s) FROM table`** picks columns; `*` selects all.
- **`AS`** creates column aliases for display. **`DISTINCT`** returns only unique values.
- **`WHERE`** filters rows using: `=`, `!=`, `>`, `<`, `>=`, `<=` for comparisons; `AND`, `OR`, `NOT` for combining; `BETWEEN` for ranges; `IN` for lists; `LIKE` with `%` and `_` for patterns; `IS NULL` / `IS NOT NULL` for missing values.
- **Never use `= NULL`** — always use `IS NULL`.
- **`ORDER BY col ASC|DESC`** sorts results; list multiple columns to break ties.
- **`LIMIT n OFFSET m`** caps results and enables pagination. Always pair with `ORDER BY`.
