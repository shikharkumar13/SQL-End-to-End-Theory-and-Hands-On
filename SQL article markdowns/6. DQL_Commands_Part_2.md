# Day 6: DQL Part 2 — Aggregation and Grouping

*A beginner's guide to databases for aspiring data analysts and data scientists*

Welcome to Day 6. Yesterday you learned to retrieve and filter rows. That's reading data. Today you learn to *summarise* it — to collapse many rows into computed results that tell you something meaningful. How many orders did each customer place? What's the total revenue? Which product category has the highest average price? These are the questions that drive real analysis, and they all rely on **aggregation**.

By the end of today you'll know the five aggregate functions, how to group data into categories with `GROUP BY`, and how to filter those groups with `HAVING` — the one clause that trips up almost every beginner until it clicks. When it does, an entire class of analytical questions becomes straightforward to write.

All queries run on the `online_store` database from Days 3 and 4, in MySQL.

---

## 1. The shift from reading rows to computing across them

On Day 5, every query returned one output row per input row (possibly filtered down). `SELECT product_name, price FROM products WHERE price > 1000` returns seven rows — one per matching product. The shape of the output mirrors the shape of the input.

Aggregation breaks that one-to-one mapping. An **aggregate function** takes a whole column's worth of values (or a group of them) and computes a single result:

```sql
SELECT COUNT(*) FROM orders;
```

The `orders` table has 15 rows, but this query returns exactly **one** row: the number 15. Many rows collapsed into one computed value. That is aggregation.

The five aggregate functions in SQL cover almost every summary an analyst needs:

| Function | What it computes |
|----------|-----------------|
| `COUNT()` | Number of rows |
| `SUM()` | Total of all values |
| `AVG()` | Mean (average) of all values |
| `MIN()` | Smallest value |
| `MAX()` | Largest value |

We'll cover each one, then learn how to compute them per group — which is where they become truly powerful.

---

## 2. COUNT — counting rows

`COUNT` is the most-used aggregate in SQL. It comes in three flavours that behave differently, and understanding the difference matters.

**`COUNT(*)`** counts every row in the result, including rows with `NULL` values:

```sql
SELECT COUNT(*) AS total_customers
FROM customers;
```

Returns **10** — all ten rows in the customers table, regardless of what any column contains.

**`COUNT(column)`** counts only the rows where that specific column is **not** `NULL`:

```sql
SELECT COUNT(city) AS customers_with_city
FROM customers;
```

Returns **9** — Noah Brown's city is `NULL`, so his row is excluded. This is the `NULL` awareness built into every aggregate function: `NULL` values are silently skipped in all aggregate calculations. This is often exactly what you want (don't count unknowns), but it's essential to know it's happening.

To see both side by side and understand the difference immediately:

```sql
SELECT
    COUNT(*)    AS total_customers,
    COUNT(city) AS customers_with_city
FROM customers;
```

| total_customers | customers_with_city |
|-----------------|---------------------|
| 10 | 9 |

The gap between these two numbers tells you how many rows have a `NULL` city — a useful data-quality check.

**`COUNT(DISTINCT column)`** counts only unique non-NULL values:

```sql
SELECT COUNT(DISTINCT country) AS unique_countries
FROM customers;
```

Returns **4** — India, Pakistan, UK, and USA. Even though India appears seven times in the customers table, it's counted once.

## 3. SUM, AVG, MIN, MAX

The remaining four aggregates work on numeric (or date) columns and follow the same `NULL`-skipping rule.

**`SUM`** adds all values in a column. This is how you calculate total revenue from the `order_items` table. Revenue per line is `quantity × unit_price`, and `SUM` adds them all up:

```sql
SELECT SUM(quantity * unit_price) AS total_revenue
FROM order_items;
```

Returns **₹48,463** — the total value of all 25 order-item rows across our 15 orders. Note that you can pass an expression like `quantity * unit_price` directly into an aggregate; you don't need to store the computed value first.

**`AVG`** computes the mean. The average price across all 15 of our products:

```sql
SELECT AVG(price) AS avg_product_price
FROM products;
```

Returns **₹1,555.67**. MySQL's `AVG` returns a precise decimal.

**`MIN` and `MAX`** return the smallest and largest values. On numbers, that's intuitive:

```sql
SELECT
    MIN(price) AS cheapest,
    MAX(price) AS most_expensive
FROM products;
```

| cheapest | most_expensive |
|----------|----------------|
| 349.00 | 7999.00 |

The Ceramic Coffee Mug (₹349) is cheapest; the Noise-Cancelling Headphones (₹7,999) are most expensive.

`MIN` and `MAX` work on dates too — a very useful fact. The earliest and latest orders placed in our database:

```sql
SELECT
    MIN(order_date) AS first_order,
    MAX(order_date) AS last_order
FROM orders;
```

| first_order | last_order |
|-------------|------------|
| 2024-01-10 | 2024-06-15 |

Our data spans exactly six months of orders.

**Multiple aggregates in one query.** You can combine as many as you need in a single `SELECT`:

```sql
SELECT
    COUNT(*)            AS total_products,
    ROUND(AVG(price),2) AS avg_price,
    MIN(price)          AS min_price,
    MAX(price)          AS max_price,
    SUM(stock_quantity) AS total_units_in_stock
FROM products;
```

| total_products | avg_price | min_price | max_price | total_units_in_stock |
|----------------|-----------|-----------|-----------|----------------------|
| 15 | 1555.67 | 349.00 | 7999.00 | 1908 |

`ROUND(AVG(price), 2)` rounds the average to two decimal places — a practical touch for displaying currency. `ROUND(value, decimal_places)` is one of the MySQL numeric functions you'll use constantly.

## 4. GROUP BY — computing per group

All the queries above collapse the entire table into a single summary row. Useful, but limited. The real power comes from computing aggregates *per group* — one summary row per category, per customer, per status, per month. That's what `GROUP BY` does.

`GROUP BY` splits the rows into groups based on one (or more) columns, and then an aggregate function is applied to each group separately.

**How many orders has each customer placed?** Group the `orders` table by `customer_id` and count the rows in each group:

```sql
SELECT
    customer_id,
    COUNT(*) AS orders_placed
FROM orders
GROUP BY customer_id
ORDER BY orders_placed DESC;
```

| customer_id | orders_placed |
|-------------|---------------|
| 1 | 3 |
| 2 | 2 |
| 3 | 2 |
| 4 | 2 |
| 5 | 2 |
| 6 | 1 |
| 7 | 1 |
| 8 | 1 |
| 9 | 1 |

Customer 1 placed 3 orders — our most active customer. Note that `customer_id 10` (Noah Brown) doesn't appear at all — he placed no orders, so there's no row in `orders` to count. We'll handle absent rows like Noah's when we reach `LEFT JOIN` on Day 7.

**Count orders by status** — how many orders are in each state?

```sql
SELECT
    status,
    COUNT(*) AS order_count
FROM orders
GROUP BY status;
```

| status | order_count |
|--------|-------------|
| Completed | 11 |
| Pending | 2 |
| Cancelled | 2 |

**How many products fall in each category?** And what's the average price per category?

```sql
SELECT
    category_id,
    COUNT(*)           AS product_count,
    ROUND(AVG(price),2) AS avg_price,
    MIN(price)         AS min_price,
    MAX(price)         AS max_price
FROM products
GROUP BY category_id
ORDER BY avg_price DESC;
```

| category_id | product_count | avg_price | min_price | max_price |
|-------------|---------------|-----------|-----------|-----------|
| 1 | 4 | 3149.00 | 799.00 | 7999.00 |
| 3 | 3 | 1265.67 | 499.00 | 1799.00 |
| 4 | 3 | 882.33 | 349.00 | 1599.00 |
| 2 | 3 | 899.00 | 599.00 | 1199.00 |
| 5 | 2 | 799.00 | 599.00 | 999.00 |

Electronics (category 1) has an average price of ₹3,149 — driven up by the Headphones. Sports (category 5) is the most affordable on average.

**Revenue and items per order** — using `order_items`:

```sql
SELECT
    order_id,
    COUNT(*)                        AS line_items,
    SUM(quantity)                   AS total_units,
    SUM(quantity * unit_price)      AS order_total
FROM order_items
GROUP BY order_id
ORDER BY order_total DESC;
```

| order_id | line_items | total_units | order_total |
|----------|------------|-------------|-------------|
| 12 | 2 | 2 | 9198.00 |
| 2 | 1 | 1 | 7999.00 |
| 4 | 2 | 2 | 3798.00 |
| 11 | 1 | 2 | 3598.00 |
| 10 | 2 | 2 | 3298.00 |

Order 12 is our highest-value order at ₹9,198, containing a Headphones and a Data Science Handbook.

## 5. The SELECT rule with GROUP BY

Here is the single most important technical rule of `GROUP BY`, the one that causes the most beginner errors:

> **Every column in your `SELECT` must either appear in the `GROUP BY` clause, or be wrapped inside an aggregate function.**

This rule exists for a logical reason. When MySQL groups rows, it collapses many rows into one. A `GROUP BY category_id` might collapse 4 products into 1 group row. For `category_id`, it's clear what to show — every row in the group has the same `category_id`. But what about `product_name`? That column has *four different values* in the Electronics group. MySQL can't decide which one to show. The rule stops you from asking an unanswerable question.

```sql
-- ✕ WRONG — product_name is not in GROUP BY and not aggregated
SELECT category_id, product_name, COUNT(*)
FROM products
GROUP BY category_id;

-- ✓ RIGHT — aggregate product_name instead
SELECT category_id, COUNT(*) AS product_count
FROM products
GROUP BY category_id;

-- ✓ RIGHT — add product_name to GROUP BY (groups by both columns)
SELECT category_id, product_name, COUNT(*)
FROM products
GROUP BY category_id, product_name;
```

MySQL 8's default `sql_mode` includes `ONLY_FULL_GROUP_BY`, which enforces this rule strictly and throws an error if you violate it. This is correct behaviour — it protects you from ambiguous, unpredictable results.

## 6. GROUP BY with multiple columns

You can group by more than one column. MySQL creates one group per unique *combination* of all the listed columns.

Count the number of orders by status and by the month they were placed:

```sql
SELECT
    MONTH(order_date)     AS order_month,
    status,
    COUNT(*)              AS order_count
FROM orders
GROUP BY MONTH(order_date), status
ORDER BY order_month, status;
```

`MONTH()` is a MySQL date function that extracts the month number from a date. This query produces a breakdown of every status that appeared in each month of our data:

| order_month | status | order_count |
|-------------|--------|-------------|
| 1 | Completed | 2 |
| 2 | Completed | 2 |
| 2 | Pending | 1 |
| 3 | Cancelled | 1 |
| 3 | Completed | 2 |
| 4 | Completed | 2 |
| 4 | Pending | 1 |
| 5 | Completed | 2 |
| 6 | Cancelled | 1 |
| 6 | Completed | 1 |

You can see January had 2 completed orders, March had 1 cancelled and 2 completed, and so on. This kind of breakdown — metric by time and category — is the structure behind almost every business dashboard.

MySQL's date functions you'll use most often in `GROUP BY`:

| Function | Returns | Example result |
|----------|---------|----------------|
| `MONTH(date)` | Month number 1–12 | `3` |
| `YEAR(date)` | Four-digit year | `2024` |
| `MONTHNAME(date)` | Month name | `'March'` |
| `DAYNAME(date)` | Day of week | `'Tuesday'` |
| `QUARTER(date)` | Quarter 1–4 | `1` |
| `DATE_FORMAT(date, fmt)` | Formatted string | `'2024-03'` |

The most flexible is `DATE_FORMAT`. To group orders by year and month as a readable label:

```sql
SELECT
    DATE_FORMAT(order_date, '%Y-%m') AS year_month,
    COUNT(*)                          AS orders_placed
FROM orders
GROUP BY DATE_FORMAT(order_date, '%Y-%m')
ORDER BY year_month;
```

| year_month | orders_placed |
|------------|---------------|
| 2024-01 | 2 |
| 2024-02 | 3 |
| 2024-03 | 3 |
| 2024-04 | 3 |
| 2024-05 | 2 |
| 2024-06 | 2 |

Month-by-month order counts — the kind of trend table that goes straight into a management report.

## 7. HAVING — filtering groups

`WHERE` filters individual rows before grouping. But what if you want to filter based on the result of an aggregate — for example, "show me only customers who placed more than one order"? You can't use `WHERE` for that, because `COUNT(*)` doesn't exist yet when `WHERE` runs. That's what **`HAVING`** is for.

`HAVING` filters *groups* after the aggregate has been computed. It is to `GROUP BY` what `WHERE` is to `FROM`.

**Find customers who placed more than one order:**

```sql
SELECT
    customer_id,
    COUNT(*) AS orders_placed
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > 1
ORDER BY orders_placed DESC;
```

| customer_id | orders_placed |
|-------------|---------------|
| 1 | 3 |
| 2 | 2 |
| 3 | 2 |
| 4 | 2 |
| 5 | 2 |

Five customers have ordered more than once. The four customers with only one order, and Noah with none, are excluded.

**Find product categories whose average price exceeds ₹1,000:**

```sql
SELECT
    category_id,
    ROUND(AVG(price), 2) AS avg_price
FROM products
GROUP BY category_id
HAVING AVG(price) > 1000
ORDER BY avg_price DESC;
```

| category_id | avg_price |
|-------------|-----------|
| 1 | 3149.00 |
| 3 | 1265.67 |

Only Electronics (₹3,149 average) and Clothing (₹1,265.67 average) pass the threshold. Books, Home & Kitchen, and Sports all have averages below ₹1,000.

**Find orders whose total value exceeds ₹3,000:**

```sql
SELECT
    order_id,
    SUM(quantity * unit_price) AS order_total
FROM order_items
GROUP BY order_id
HAVING SUM(quantity * unit_price) > 3000
ORDER BY order_total DESC;
```

| order_id | order_total |
|----------|-------------|
| 12 | 9198.00 |
| 2 | 7999.00 |
| 4 | 3798.00 |
| 11 | 3598.00 |
| 10 | 3298.00 |

Five of our 15 orders exceed ₹3,000 in value.

## 8. WHERE vs HAVING — the crucial distinction

This is the concept that trips up almost every SQL beginner, so let's be precise.

| | WHERE | HAVING |
|---|-------|--------|
| **Filters** | Individual rows | Groups of rows |
| **Runs** | Before `GROUP BY` | After `GROUP BY` |
| **Can use aggregates?** | No | Yes |
| **Can use plain columns?** | Yes | Yes |

The updated execution order with all today's clauses added:

```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

`WHERE` runs early — it narrows the rows that enter the grouping step. `HAVING` runs late — it trims the groups that come out of it. They filter at different points in the pipeline.

**You can and often should use both together.** Here's a query that asks: *"Among completed orders only, which customers placed more than one?"*

```sql
SELECT
    customer_id,
    COUNT(*) AS completed_orders
FROM orders
WHERE status = 'Completed'
GROUP BY customer_id
HAVING COUNT(*) > 1
ORDER BY completed_orders DESC;
```

Step by step through the execution order:
1. **FROM** — start with all 15 rows from `orders`.
2. **WHERE** `status = 'Completed'` — keep only the 11 completed orders (4 cancelled/pending are removed before grouping).
3. **GROUP BY** `customer_id` — group the 11 remaining rows by customer.
4. **HAVING** `COUNT(*) > 1` — keep only groups with more than one row.
5. **SELECT** — return `customer_id` and the count.
6. **ORDER BY** — sort descending.

| customer_id | completed_orders |
|-------------|-----------------|
| 1 | 3 |
| 3 | 2 |

Only customers 1 and 3 have more than one *completed* order. Customer 2 has two orders total, but one is cancelled — so it's filtered out by `WHERE status = 'Completed'` before grouping, leaving customer 2 with only one completed order.

This is the `WHERE` vs `HAVING` distinction made concrete: `WHERE` filters *before* the counts happen; `HAVING` filters *after*. Same-looking words, completely different points in the pipeline.

> **A practical rule of thumb.** If your filter condition doesn't involve an aggregate function, use `WHERE` — it's more efficient because it reduces the rows that need to be grouped. If your condition *does* involve an aggregate (`COUNT`, `SUM`, `AVG`, etc.), you must use `HAVING`.

## 9. Putting it all together — a real analyst query

Let's close with a query that combines everything from today. The question:

> *"For each product, how many times has it been ordered, and what is its total revenue? Show only products that have appeared in at least 2 order lines, ranked by total revenue."*

```sql
SELECT
    product_id,
    COUNT(*)                     AS times_ordered,
    SUM(quantity)                AS total_units_sold,
    SUM(quantity * unit_price)   AS total_revenue
FROM order_items
GROUP BY product_id
HAVING COUNT(*) >= 2
ORDER BY total_revenue DESC;
```

Let's trace the execution:
1. **FROM** `order_items` — 25 rows.
2. **GROUP BY** `product_id` — groups collapse into one row per product (15 distinct products appear).
3. **HAVING** `COUNT(*) >= 2` — keep only products that appear in 2 or more order lines.
4. **SELECT** — compute the four columns for each surviving group.
5. **ORDER BY** `total_revenue DESC` — sort highest revenue first.

| product_id | times_ordered | total_units_sold | total_revenue |
|------------|---------------|------------------|---------------|
| 4 | 2 | 2 | 15998.00 |
| 1 | 3 | 4 | 3197.00 |
| 5 | 3 | 4 | 2397.00 |
| 8 | 2 | 5 | 2495.00 |
| 9 | 2 | 3 | 5397.00 |
| 2 | 2 | 2 | 4998.00 |
| 6 | 2 | 3 | 2398.00 |
| 11 | 2 | 3 | 2097.00 |
| 14 | 2 | 2 | 1998.00 |

Product 4 (Noise-Cancelling Headphones) tops the revenue chart at ₹15,998 — two orders but at ₹7,999 each, it generates more revenue than anything else. Product 1 (Wireless Mouse) appeared in 3 separate order lines — the most frequently ordered item. Products 13 (Non-Stick Frying Pan) and 15 (Resistance Bands Set) don't appear here at all: the Frying Pan was never ordered, and the Bands Set appears in only one order line — filtered out by `HAVING COUNT(*) >= 2`.

This kind of product performance table — sorted by revenue, with frequency and volume alongside — is a staple of e-commerce analytics, and you've just built it from scratch.

## Where we're headed

Today you learned to compute across rows. But all our queries so far draw from a single table. The real analytical questions in `online_store` span multiple tables at once: which *category name* (not ID) has the highest revenue? What are the names of customers who placed the most orders? Which products have never been sold?

These questions require combining data from two or more tables in a single query — and that's exactly what **Day 7** covers: `JOIN`. Joins are the most transformative concept in SQL, and because you now understand keys (Day 1), schema design (Day 2), and aggregation (today), they'll land clearly.

---

### Quick recap of Day 6

- **Aggregate functions** collapse many rows into one computed value: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`.
- `COUNT(*)` counts all rows; `COUNT(column)` skips `NULL`s; `COUNT(DISTINCT col)` counts unique values.
- All aggregates skip `NULL` values — know when this matters.
- `ROUND(value, n)` rounds to `n` decimal places; pair it with `AVG` for clean output.
- **`GROUP BY`** splits rows into groups and applies the aggregate to each group separately.
- Every column in `SELECT` must be in `GROUP BY` or wrapped in an aggregate — no exceptions.
- MySQL date functions `MONTH()`, `YEAR()`, `DATE_FORMAT()` let you group by time periods.
- **`HAVING`** filters groups after aggregation — it's the only place to use aggregate conditions.
- `WHERE` filters rows *before* grouping (no aggregates allowed); `HAVING` filters groups *after* (aggregates required for the filter itself). Use both together when you need to.
- Execution order: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`.
