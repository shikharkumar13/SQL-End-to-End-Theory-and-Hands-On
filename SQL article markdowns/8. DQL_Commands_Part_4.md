# Day 8: DQL Part 4 — Subqueries, CTEs, and Conditional Logic

*A beginner's guide to databases for aspiring data analysts and data scientists*

Welcome to Day 8. The tools you have — filtering, aggregating, joining — can already answer a lot of business questions. Today you get three more that unlock a different kind of power: the ability to **ask a question using the answer to another question**. Subqueries let you nest one query inside another. CTEs let you name those intermediate results so your code stays readable. And CASE expressions let you embed conditional logic — if/else thinking — directly inside a query.

Together they let you express complex business reasoning in SQL rather than pre-processing data in Python or a spreadsheet first.

All queries are MySQL-compatible and run on the `online_store` database from Day 4.

---

## 1. Queries inside queries

A **subquery** (also called an inner query or nested query) is a `SELECT` statement written inside another SQL statement. The outer query uses the subquery's result as though it were a value, a list, or a table.

The simplest way to see why this matters: suppose you want the products that cost more than the overall average price. You don't know that average yet — it's a number the database has to compute first. A subquery lets you compute it inline:

```sql
SELECT product_name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products);
```

The inner `SELECT AVG(price) FROM products` runs first, returns `1555.67`, and the outer query then uses that value to filter. No temp table, no multi-step process — the computation and the filtering happen in one statement.

Subqueries come in three flavours based on what they return: a **single value** (scalar), a **list of values** (multi-row), or a **table** (derived). We'll cover each one, then meet CTEs — the cleaner, more readable alternative to deeply nested subqueries.

---

## 2. Scalar subqueries — single values in WHERE and SELECT

A **scalar subquery** returns exactly one column and one row: a single value. You can use it anywhere a single value is expected — in `WHERE`, in `SELECT`, even in `ORDER BY`.

**Filter with a computed threshold.** The query above is the classic example. The inner query computes the average; the outer query filters against it:

```sql
SELECT product_name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products)
ORDER BY price DESC;
```

Our average price is ₹1,555.67, so this returns 4 products:

| product_name | price |
|--------------|-------|
| Noise-Cancelling Headphones | 7999.00 |
| Mechanical Keyboard | 2499.00 |
| Denim Jeans | 1799.00 |
| Non-Stick Frying Pan | 1599.00 |

**Scalar subquery in SELECT.** You can put a scalar subquery directly in the `SELECT` clause to add a computed column to every row:

```sql
SELECT
    product_name,
    price,
    (SELECT ROUND(AVG(price), 2) FROM products) AS overall_avg,
    price - (SELECT ROUND(AVG(price), 2) FROM products) AS diff_from_avg
FROM products
ORDER BY diff_from_avg DESC;
```

This adds `overall_avg` (₹1,555.67) and the gap from it to every product row. The Noise-Cancelling Headphones sit ₹6,443.33 above average; the Ceramic Coffee Mug sits ₹1,206.67 below. Notice the scalar subquery repeats — we'll replace this repetition with a CTE in Section 6.

> **A scalar subquery must return exactly one row and one column.** If it returns more than one row, MySQL throws an error. If it returns no rows, the result is `NULL`. Keep this constraint in mind when writing them.

## 3. Multi-row subqueries — lists with IN and NOT IN

A **multi-row subquery** returns a column of multiple values. You use it with `IN`, `NOT IN`, `ANY`, or `ALL` — operators that compare a value against a whole list.

**Find customers who have placed at least one order**, using the list of `customer_id` values that appear in the `orders` table:

```sql
SELECT first_name, last_name
FROM customers
WHERE customer_id IN (
    SELECT DISTINCT customer_id FROM orders
);
```

The inner query returns 9 distinct `customer_id` values (everyone except Noah Brown). The outer query returns the 9 customers whose ID appears in that list — effectively a join, but written as a filter.

**The inverse — customers who have never ordered**, using `NOT IN`:

```sql
SELECT first_name, last_name
FROM customers
WHERE customer_id NOT IN (
    SELECT DISTINCT customer_id FROM orders
);
```

Returns one row: Noah Brown. This is the same result as the `LEFT JOIN ... IS NULL` pattern from Day 7, but expressed differently. Both approaches are valid — use whichever reads more clearly to you.

**Find products that were ordered in January 2024.** The inner query identifies the relevant `product_id` values; the outer query retrieves their details:

```sql
SELECT product_name, price
FROM products
WHERE product_id IN (
    SELECT DISTINCT oi.product_id
    FROM order_items oi
    JOIN orders o ON oi.order_id = o.order_id
    WHERE o.order_date BETWEEN '2024-01-01' AND '2024-01-31'
);
```

January 2024 had two orders. The inner query finds the products on those orders (product IDs 1, 4, and 5), and the outer query returns their names and prices:

| product_name | price |
|--------------|-------|
| Wireless Mouse | 799.00 |
| Noise-Cancelling Headphones | 7999.00 |
| SQL Fundamentals | 599.00 |

> **`NOT IN` and NULL — the same trap as Day 5.** If the subquery result contains even one `NULL` value, `NOT IN` returns no rows at all. This happens silently and is very hard to debug. Whenever using `NOT IN` with a subquery, always add `WHERE column IS NOT NULL` inside the inner query to exclude NULLs from the list.

## 4. Correlated subqueries — referencing the outer query

A **correlated subquery** is a subquery that references a column from the outer query. Unlike the subqueries above (which run once and produce a result), a correlated subquery runs once for every row processed by the outer query. The inner query "sees" the current outer row.

**Find products priced above the average of their own category.** This is a question the previous subquery types can't answer in a single step — you need to compare each product to the average of *its specific category*, not the overall average. A correlated subquery solves it:

```sql
SELECT
    p.product_name,
    p.price,
    cat.category_name
FROM products p
JOIN categories cat ON p.category_id = cat.category_id
WHERE p.price > (
    SELECT AVG(p2.price)
    FROM products p2
    WHERE p2.category_id = p.category_id
)
ORDER BY cat.category_name, p.price DESC;
```

The inner query `WHERE p2.category_id = p.category_id` refers to `p.category_id` — a column from the outer query's current row. For each product, it computes the average for *that product's category* and compares. The result contains 6 products:

| product_name | price | category_name |
|--------------|-------|---------------|
| Noise-Cancelling Headphones | 7999.00 | Electronics |
| Data Science Handbook | 1199.00 | Books |
| Denim Jeans | 1799.00 | Clothing |
| Hooded Sweatshirt | 1499.00 | Clothing |
| Non-Stick Frying Pan | 1599.00 | Home & Kitchen |
| Yoga Mat | 999.00 | Sports |

Each of these is the premium option in its category. Notice the Wireless Mouse (₹799) is excluded even though it seems reasonably priced — it's below the Electronics average of ₹3,149, dragged upward by the Headphones.

**EXISTS — the efficient alternative for "does a match exist?"**

`EXISTS` is a close sibling of correlated subqueries. Instead of returning a value or list, it simply checks whether the inner query returns any rows at all, and short-circuits as soon as one match is found.

```sql
-- Customers who have placed at least one order (using EXISTS)
SELECT first_name, last_name
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.customer_id
);
```

`SELECT 1` is the convention inside `EXISTS` — we don't care what the inner query returns, only whether it returns anything. For each customer, MySQL checks: does any row in `orders` match this `customer_id`? If yes, the customer is included. This returns the same 9 customers as the `IN` version above.

`EXISTS` tends to be more efficient than `IN` when the inner query would return a large result set, because it stops searching after the first match rather than building the complete list first.

## 5. Derived tables — subqueries as virtual tables

A **derived table** is a subquery placed in the `FROM` clause, where it acts as a temporary, unnamed table for the outer query. MySQL requires you to give it an alias.

**Find customers whose total spend exceeds the average customer spend.** This requires two steps that can't easily be written in one level: first compute each customer's total, then compare those totals to their own average. A derived table handles the first step:

```sql
SELECT
    first_name,
    last_name,
    total_spent
FROM (
    SELECT
        c.customer_id,
        c.first_name,
        c.last_name,
        SUM(oi.quantity * oi.unit_price) AS total_spent
    FROM customers c
    JOIN orders      o  ON c.customer_id = o.customer_id
    JOIN order_items oi ON o.order_id    = oi.order_id
    GROUP BY c.customer_id, c.first_name, c.last_name
) AS customer_totals
WHERE total_spent > (
    SELECT AVG(total_spent)
    FROM (
        SELECT
            c.customer_id,
            SUM(oi.quantity * oi.unit_price) AS total_spent
        FROM customers c
        JOIN orders      o  ON c.customer_id = o.customer_id
        JOIN order_items oi ON o.order_id    = oi.order_id
        GROUP BY c.customer_id
    ) AS sub
)
ORDER BY total_spent DESC;
```

This works, but look at the problem: the customer-spend calculation appears **twice** — once as the main derived table and once inside the scalar subquery. The code is repetitive, hard to read, and difficult to maintain. There's a better way.

## 6. CTEs — naming your subqueries

A **CTE (Common Table Expression)** lets you give a subquery a name and reference it like a regular table. You define it with the `WITH` keyword before the main query:

```sql
WITH cte_name AS (
    SELECT ...  -- this is your named subquery
)
SELECT ... FROM cte_name WHERE ...;
```

Here is the derived-table query above rewritten with a CTE:

```sql
WITH customer_totals AS (
    SELECT
        c.customer_id,
        c.first_name,
        c.last_name,
        SUM(oi.quantity * oi.unit_price) AS total_spent
    FROM customers c
    JOIN orders      o  ON c.customer_id = o.customer_id
    JOIN order_items oi ON o.order_id    = oi.order_id
    GROUP BY c.customer_id, c.first_name, c.last_name
)
SELECT first_name, last_name, total_spent
FROM customer_totals
WHERE total_spent > (SELECT AVG(total_spent) FROM customer_totals)
ORDER BY total_spent DESC;
```

| first_name | last_name | total_spent |
|------------|-----------|-------------|
| Aarav | Sharma | 15193.00 |
| Diya | Patel | 9498.00 |
| Rohan | Mehta | 6594.00 |

The average customer spend (among the 9 who placed orders) is ₹5,384.78. Three customers exceed it.

The improvement over the derived-table version is significant:

- The `customer_totals` CTE is defined **once** but referenced **twice** — in the outer `FROM` and in the `AVG` subquery inside `WHERE`. No duplication.
- The structure reads top-to-bottom like a recipe: "first compute customer totals, then find those above average." Each step is visible and named.
- You can test the CTE independently during development by temporarily running just `SELECT * FROM customer_totals` on its own — you can't do this with a nested derived table.

CTEs are not faster than equivalent subqueries (MySQL optimises them similarly), but they make complex queries dramatically easier to read, write, and debug. For any query with more than one level of nesting, a CTE is almost always the right choice.

## 7. Multiple CTEs — chaining WITH clauses

You can define multiple CTEs in one `WITH` clause by separating them with commas. Each CTE can reference the CTEs defined before it. This lets you build complex multi-step logic as a clean sequence of named steps.

**Monthly performance with percentage of peak.** Three steps: compute monthly revenue, find the peak, then combine both:

```sql
WITH monthly_revenue AS (
    SELECT
        DATE_FORMAT(o.order_date, '%Y-%m')   AS month,
        COUNT(DISTINCT o.order_id)            AS orders_placed,
        SUM(oi.quantity * oi.unit_price)      AS revenue
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY DATE_FORMAT(o.order_date, '%Y-%m')
),
peak AS (
    SELECT MAX(revenue) AS peak_revenue
    FROM monthly_revenue
)
SELECT
    mr.month,
    mr.orders_placed,
    mr.revenue,
    ROUND(mr.revenue / p.peak_revenue * 100, 1) AS pct_of_peak
FROM monthly_revenue mr
CROSS JOIN peak p
ORDER BY mr.month;
```

| month | orders_placed | revenue | pct_of_peak |
|-------|---------------|---------|-------------|
| 2024-01 | 2 | 10196.00 | 100.0 |
| 2024-02 | 3 | 9492.00 | 93.1 |
| 2024-03 | 3 | 6490.00 | 63.7 |
| 2024-04 | 3 | 8394.00 | 82.3 |
| 2024-05 | 2 | 10196.00 | 100.0 |
| 2024-06 | 2 | 3695.00 | 36.2 |

January and May share the peak at ₹10,196 — both driven by single large orders containing Noise-Cancelling Headphones (₹7,999). The `CROSS JOIN peak` attaches the single-row result to every monthly row so the percentage can be calculated.

Notice how readable the query is top to bottom: "first build monthly_revenue, then find the peak, then display the result with context." Without CTEs, this logic would require three levels of nesting.

## 8. CASE expressions — conditional logic in SQL

A **CASE expression** evaluates conditions and returns a value, much like an if/else statement. It can appear in `SELECT`, `WHERE`, `ORDER BY`, and crucially inside aggregate functions. There are two forms.

**Simple CASE** — matches a column against a set of exact values:

```sql
SELECT
    order_id,
    status,
    CASE status
        WHEN 'Completed'  THEN 'Revenue Confirmed'
        WHEN 'Pending'    THEN 'Awaiting Fulfilment'
        WHEN 'Cancelled'  THEN 'No Revenue'
        ELSE 'Unknown'
    END AS business_label
FROM orders
ORDER BY order_id;
```

Each status gets a business-friendly label. The `ELSE` clause is a safety net for any value not listed in the `WHEN`s — always include it unless you're certain every value is covered.

**Searched CASE** — evaluates independent conditions in order, returning the first one that is true. This is the more flexible and more commonly used form:

```sql
SELECT
    product_name,
    price,
    CASE
        WHEN price < 500   THEN 'Budget'
        WHEN price < 1000  THEN 'Mid-Range'
        WHEN price < 3000  THEN 'Premium'
        ELSE                    'Luxury'
    END AS price_tier
FROM products
ORDER BY price;
```

The `WHEN` clauses are evaluated top to bottom and the first match wins — so a price of ₹499 matches `price < 500` immediately, without ever checking the higher thresholds. Our 15 products break down as: 2 Budget, 6 Mid-Range, 6 Premium, 1 Luxury (the Noise-Cancelling Headphones).

**CASE with NULL.** Use `CASE` alongside `COALESCE` — a function that returns the first non-NULL argument — to handle missing values cleanly:

```sql
SELECT
    first_name,
    last_name,
    COALESCE(city, 'City not provided') AS city_display
FROM customers;
```

Noah Brown's `NULL` city now shows as 'City not provided' instead of a blank cell — a practical touch for any report intended for non-technical readers.

## 9. CASE inside aggregate functions — conditional aggregation

The most powerful use of CASE is inside aggregate functions, enabling you to compute multiple filtered aggregates in a single pass over the data — often called a **pivot** or **conditional aggregation**.

**Count orders by status in one row**, instead of returning a row per status:

```sql
SELECT
    COUNT(*)                                            AS total_orders,
    COUNT(CASE WHEN status = 'Completed'  THEN 1 END)  AS completed,
    COUNT(CASE WHEN status = 'Pending'    THEN 1 END)  AS pending,
    COUNT(CASE WHEN status = 'Cancelled'  THEN 1 END)  AS cancelled
FROM orders;
```

| total_orders | completed | pending | cancelled |
|--------------|-----------|---------|-----------|
| 15 | 11 | 2 | 2 |

Each `COUNT(CASE WHEN ... THEN 1 END)` returns `1` for matching rows and `NULL` for non-matching rows — and since `COUNT` ignores `NULL`, it effectively counts only the matching rows. The whole table is scanned once, but you get three filtered counts simultaneously.

**Revenue split by order status — confirmed, at-risk, and lost:**

```sql
SELECT
    SUM(CASE WHEN o.status = 'Completed'
             THEN oi.quantity * oi.unit_price ELSE 0 END) AS confirmed_revenue,
    SUM(CASE WHEN o.status = 'Pending'
             THEN oi.quantity * oi.unit_price ELSE 0 END) AS pending_revenue,
    SUM(CASE WHEN o.status = 'Cancelled'
             THEN oi.quantity * oi.unit_price ELSE 0 END) AS lost_revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id;
```

| confirmed_revenue | pending_revenue | lost_revenue |
|-------------------|-----------------|--------------|
| 39969.00 | 5996.00 | 2498.00 |

₹39,969 is confirmed (82.5% of total revenue). ₹5,996 is in pending orders that might still complete. ₹2,498 is in cancelled orders — the real lost revenue. These three numbers in one query are exactly what a finance report would want. Running three separate `WHERE` queries and manually combining results would give the same answer but take three times as long to write.

## 10. Putting it all together — the customer segmentation report

Let's bring everything from today into one coherent query that answers a genuine business question:

> *"Produce a full customer report showing each customer's total orders, their confirmed (completed) and cancelled order counts, their lifetime spend, and their customer segment — including inactive customers with zero spend."*

```sql
WITH customer_summary AS (
    SELECT
        c.customer_id,
        c.first_name,
        c.last_name,
        COUNT(DISTINCT o.order_id)                                      AS total_orders,
        COUNT(CASE WHEN o.status = 'Completed'  THEN 1 END)            AS completed_orders,
        COUNT(CASE WHEN o.status = 'Cancelled'  THEN 1 END)            AS cancelled_orders,
        COALESCE(SUM(oi.quantity * oi.unit_price), 0)                   AS lifetime_spend
    FROM customers c
    LEFT JOIN orders      o   ON c.customer_id = o.customer_id
    LEFT JOIN order_items oi  ON o.order_id    = oi.order_id
    GROUP BY c.customer_id, c.first_name, c.last_name
)
SELECT
    first_name,
    last_name,
    total_orders,
    completed_orders,
    cancelled_orders,
    lifetime_spend,
    CASE
        WHEN lifetime_spend >= 10000 THEN 'VIP'
        WHEN lifetime_spend >= 5000  THEN 'Regular'
        WHEN lifetime_spend >= 1     THEN 'Occasional'
        ELSE                              'Inactive'
    END AS segment
FROM customer_summary
ORDER BY lifetime_spend DESC;
```

| first_name | last_name | total_orders | completed_orders | cancelled_orders | lifetime_spend | segment |
|------------|-----------|--------------|-----------------|-----------------|----------------|---------|
| Aarav | Sharma | 3 | 3 | 0 | 15193.00 | VIP |
| Diya | Patel | 2 | 1 | 1 | 9498.00 | Regular |
| Rohan | Mehta | 2 | 2 | 0 | 6594.00 | Regular |
| Ananya | Reddy | 2 | 1 | 0 | 5094.00 | Regular |
| Kabir | Khan | 1 | 0 | 0 | 3598.00 | Occasional |
| Vivaan | Singh | 2 | 1 | 1 | 3196.00 | Occasional |
| Ishaan | Gupta | 1 | 1 | 0 | 2794.00 | Occasional |
| Saanvi | Iyer | 1 | 1 | 0 | 1498.00 | Occasional |
| Mia | Williams | 1 | 1 | 0 | 998.00 | Occasional |
| Noah | Brown | 0 | 0 | 0 | 0.00 | Inactive |

Every technique from today does useful work in this query. The **CTE** (`customer_summary`) names the intermediate step so it can be referenced cleanly in the outer query. The **LEFT JOINs** ensure Noah appears with zeros rather than being excluded. **Conditional aggregation** with `CASE` splits the order count into completed and cancelled without extra joins. **`COALESCE`** converts Noah's `NULL` lifetime spend to `0`. And the **searched CASE** in the outer `SELECT` assigns the business segment label.

This is the kind of query that goes directly into a business dashboard or weekly stakeholder report. You've just written it from scratch.

### Quick recap of Day 8

- A **subquery** is a `SELECT` inside another SQL statement. The outer query uses the inner result as a value, list, or table.
- **Scalar subqueries** return one value; use in `WHERE` or `SELECT`. They must return exactly one row and column.
- **Multi-row subqueries** return a column of values; use with `IN` or `NOT IN`. Always exclude NULLs from the list with `WHERE column IS NOT NULL`.
- **Correlated subqueries** reference the outer query's current row and run once per outer row — used for per-row comparisons like "above its own category average."
- **`EXISTS`** checks whether a correlated subquery returns any rows; faster than `IN` on large result sets.
- **Derived tables** are subqueries in the `FROM` clause. They work but produce hard-to-read nesting when reused.
- A **CTE** (`WITH name AS (...)`) names a subquery, allows multiple references to it, and reads top-to-bottom like a recipe. Prefer CTEs over nested derived tables.
- **Multiple CTEs** chain with commas; each can reference those defined before it.
- **`CASE`** adds if/else logic: simple form matches a column's value; searched form evaluates independent conditions in order.
- **`COALESCE(col, default)`** returns the first non-NULL argument — use it to replace NULLs in output.
- **Conditional aggregation** — `COUNT(CASE WHEN ... THEN 1 END)` and `SUM(CASE WHEN ... THEN value ELSE 0 END)` — computes multiple filtered aggregates in a single table scan.
