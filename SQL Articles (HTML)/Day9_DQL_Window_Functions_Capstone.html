01

What window functions are — and why they're different

On Day 6 you learned that GROUP BY collapses many rows into one summary row per group. That's powerful, but it destroys the individual rows in the process. Sometimes you want both: the computed result and every individual row still visible.

Window functions compute a value across a set of rows — the "window" — but return a result for every row rather than collapsing them. The rows stay intact; the computation appears alongside them as a new column.

Compare these two queries:

SQL · MySQL
-- GROUP BY: collapses rows — one result per category
SELECT category_id, AVG(price) AS avg_price
FROM products
GROUP BY category_id;

-- Window function: keeps all rows — adds avg alongside each product
SELECT product_name, price, category_id,
       AVG(price) OVER (PARTITION BY category_id) AS category_avg
FROM products;

The first query returns 5 rows (one per category). The second returns all 15 product rows, each enriched with the average price of its own category. Same computation; completely different shape. This is the defining characteristic of window functions: they compute without collapsing.

All window functions share the same OVER clause syntax:

SQL · MySQL
function_name() OVER (
    PARTITION BY column   -- divide rows into groups (optional)
    ORDER BY column       -- define order within each group (often required)
    ROWS BETWEEN ...      -- define the frame (optional, shown later)
)

PARTITION BY is like GROUP BY for the window — it defines which rows belong to the same group. ORDER BY controls the sequence within each partition. Both are optional, but whether you include them changes what the function computes.

Window functions require MySQL 8.0 or higher. They are not available in MySQL 5.x. All modern MySQL installations (8.0+) support them fully.

02

Ranking functions — ROW_NUMBER, RANK, DENSE_RANK

The three ranking functions number rows within each partition according to an ORDER BY. They differ only in how they handle ties.

**ROW_NUMBER()** assigns a unique sequential number, with no ties — even if two rows have identical values, they get different numbers:

SQL · MySQL
SELECT
    cat.category_name,
    p.product_name,
    p.price,
    ROW_NUMBER() OVER (
        PARTITION BY p.category_id
        ORDER BY p.price DESC
    ) AS price_rank
FROM products p
JOIN categories cat ON p.category_id = cat.category_id
ORDER BY cat.category_name, price_rank;

This returns all 15 products, each numbered 1 through however many products are in their category — the most expensive in each category gets rank 1. Since no two products share a price in our data, this works cleanly. ROW_NUMBER guarantees uniqueness by breaking ties arbitrarily when values are equal — useful when you need exactly one "winner" per group.

**RANK() and DENSE_RANK()** handle ties more explicitly. When rows share the same value, both give them the same rank — but they differ in what comes next:

  • **RANK()** skips numbers after a tie. Four customers tied at rank 2 means the next rank is 6.
  • **DENSE_RANK()** does not skip. Four customers tied at rank 2 are still followed by rank 3.

Our orders table has a natural tie: four customers placed exactly 2 orders each. This makes the difference concrete:

SQL · MySQL
SELECT
    c.first_name,
    c.last_name,
    COUNT(o.order_id)                                    AS orders_placed,
    RANK()       OVER (ORDER BY COUNT(o.order_id) DESC)  AS rnk,
    DENSE_RANK() OVER (ORDER BY COUNT(o.order_id) DESC)  AS dense_rnk
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.first_name, c.last_name
ORDER BY orders_placed DESC;

Use RANK when the position gaps matter (e.g. top 3 podium), DENSE_RANK when you want continuous tier numbers (e.g. bronze/silver/gold groupings).

Finding the top-N per group — one of the most common analytical patterns — pairs ROW_NUMBER inside a CTE with a WHERE rn = 1 filter in the outer query. The capstone section uses this to find the best-selling product within each category.

03

LAG and LEAD — looking at adjacent rows

LAG(column, n) returns the value from n rows before the current row in the window; LEAD(column, n) returns the value from n rows after it. Both default to n = 1. Rows where there is no predecessor or successor return NULL.

These are the functions for period-over-period comparison — the backbone of trend analysis.

Month-over-month revenue change:

SQL · MySQL
WITH monthly AS (
    SELECT
        DATE_FORMAT(o.order_date, '%Y-%m')    AS month,
        SUM(oi.quantity * oi.unit_price)       AS revenue
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY DATE_FORMAT(o.order_date, '%Y-%m')
)
SELECT
    month,
    revenue,
    LAG(revenue)  OVER (ORDER BY month) AS prev_month,
    LEAD(revenue) OVER (ORDER BY month) AS next_month,
    revenue - LAG(revenue) OVER (ORDER BY month) AS mom_change
FROM monthly
ORDER BY month;

04

Running totals and window aggregates

Standard aggregate functions (SUM, AVG, COUNT, MIN, MAX) all work as window functions when paired with OVER. Without PARTITION BY, they compute across the entire result. With ORDER BY inside OVER, they become cumulative — each row sees only the rows up to and including itself.

Cumulative (running) revenue total across all months:

SQL · MySQL
WITH monthly AS (
    SELECT
        DATE_FORMAT(o.order_date, '%Y-%m')    AS month,
        COUNT(DISTINCT o.order_id)             AS orders_placed,
        SUM(oi.quantity * oi.unit_price)       AS revenue
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY DATE_FORMAT(o.order_date, '%Y-%m')
)
SELECT
    month,
    orders_placed,
    revenue,
    SUM(revenue) OVER (
        ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM monthly
ORDER BY month;

The ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW frame clause tells MySQL: for each row, include all rows from the very first row up to and including this one. The final running total is ₹48,463 — matching the grand total we've verified throughout the series.

Always specify the frame explicitly for running totals. MySQL's default frame for SUM() OVER (ORDER BY ...) is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which handles ties in a way that may double-count when two rows share the same ORDER BY value. ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is always safe and unambiguous.

**Percentage share using SUM() OVER () with no frame.** When you omit ORDER BY from OVER, the window includes every row — giving you the grand total in every row, which is perfect for calculating percentage shares:

SQL · MySQL
SELECT
    cat.category_name,
    SUM(oi.quantity * oi.unit_price)                              AS cat_revenue,
    ROUND(
        SUM(oi.quantity * oi.unit_price) /
        SUM(SUM(oi.quantity * oi.unit_price)) OVER () * 100
    , 1)                                                           AS pct_share
FROM categories cat
JOIN products    p  ON cat.category_id = p.category_id
JOIN order_items oi ON p.product_id    = oi.product_id
GROUP BY cat.category_id, cat.category_name
ORDER BY cat_revenue DESC;

SUM(SUM(...)) OVER () — the inner SUM is the GROUP BY aggregate per category; the outer SUM(...) OVER () is the window function that totals across all category rows. This pattern is one of the most useful in analytical SQL.

05

Views — saving queries for reuse

A view is a saved SELECT query stored in the database under a name. Once created, you can query it exactly like a table. Views don't store data — they store the query definition and re-run it each time you access the view.

They are useful for three reasons: they encapsulate complex logic behind a simple name, they let you share consistent business definitions across many queries, and they can restrict which columns certain users see.

SQL · MySQL
-- Create a view that exposes product-level revenue
CREATE VIEW v_product_revenue AS
SELECT
    p.product_id,
    p.product_name,
    cat.category_name,
    COALESCE(SUM(oi.quantity * oi.unit_price), 0) AS total_revenue
FROM products p
JOIN categories  cat ON p.category_id  = cat.category_id
LEFT JOIN order_items oi ON p.product_id = oi.product_id
GROUP BY p.product_id, p.product_name, cat.category_name;

Once the view exists, querying it is no different from querying a table:

SQL · MySQL
-- Which Electronics products generated more than ₹3,000?
SELECT product_name, total_revenue
FROM v_product_revenue
WHERE category_name = 'Electronics'
  AND total_revenue > 3000
ORDER BY total_revenue DESC;

Modify a view with CREATE OR REPLACE VIEW; remove it with DROP VIEW IF EXISTS v_product_revenue. Because views re-execute the underlying query every time they're accessed, they are not a performance optimisation — they are an organisation and reuse tool.

06

Query best practices

Before the capstone, a brief checklist of habits that separate readable, maintainable SQL from query soup.

Name your columns explicitly. Avoid SELECT * in production queries. Name only what you need, give calculated columns an alias, and your query is self-documenting.

Comment your logic. A comment before each CTE or a note on a non-obvious filter takes seconds to write and saves minutes of re-reading later.

Format for structure. Align keywords, one clause per line, indent subqueries and CTEs consistently. SQL ignores whitespace; your future self does not.

Use CTEs for complexity. Any query with more than one level of logic is clearer as a chain of named CTEs than as nested subqueries. Write the steps top to bottom, like a recipe.

Filter early. WHERE runs before GROUP BY — filtering in WHERE reduces the rows that need to be grouped or joined, which is always faster than filtering after. Use HAVING only for aggregate conditions.

**Develop with LIMIT.** When writing a new query against a large table, add LIMIT 10 while you're testing. Remove it only when you're confident the logic is right.

Verify your totals. On analytical queries, sanity-check key numbers: do your group totals sum to the known grand total? Does the customer count match SELECT COUNT(*) FROM customers? Small checks catch big mistakes.

Question 1: What are the top 5 best-selling products by total revenue?

A foundational product-performance report — JOIN + GROUP BY + ORDER BY + LIMIT.

SQL · MySQL
SELECT
    p.product_name,
    cat.category_name,
    COUNT(DISTINCT oi.order_id)           AS times_ordered,
    SUM(oi.quantity)                       AS units_sold,
    SUM(oi.quantity * oi.unit_price)       AS total_revenue
FROM products p
JOIN categories  cat ON p.category_id  = cat.category_id
JOIN order_items oi  ON p.product_id   = oi.product_id
GROUP BY p.product_id, p.product_name, cat.category_name
ORDER BY total_revenue DESC
LIMIT 5;

Question 5: Rank customers by total lifetime spend and show each one's share of the top customer's spend.

Combines a CTE with RANK() and MAX() OVER () for relative performance context.

SQL · MySQL
WITH customer_spend AS (
    SELECT
        c.customer_id,
        c.first_name,
        c.last_name,
        SUM(oi.quantity * oi.unit_price)    AS lifetime_spend
    FROM customers c
    JOIN orders      o  ON c.customer_id = o.customer_id
    JOIN order_items oi ON o.order_id    = oi.order_id

SELECT
    first_name,
    last_name,
    lifetime_spend,
    RANK() OVER (ORDER BY lifetime_spend DESC)                    AS spend_rank,
    ROUND(lifetime_spend / MAX(lifetime_spend) OVER () * 100, 1) AS pct_of_top

Question 6: Which is the top-revenue product within each category?

The classic "top-N per group" pattern — ROW_NUMBER() inside a CTE, filtered in the outer query.

SQL · MySQL
WITH product_revenue AS (
    SELECT
        p.product_id,
        p.product_name,
        cat.category_name,
        SUM(oi.quantity * oi.unit_price) AS total_revenue
    FROM products p
    JOIN categories  cat ON p.category_id  = cat.category_id
    JOIN order_items oi  ON p.product_id   = oi.product_id
    GROUP BY p.product_id, p.product_name, cat.category_name
),
ranked AS (
    SELECT *,
        ROW_NUMBER() OVER (
            PARTITION BY category_name
            ORDER BY total_revenue DESC
        ) AS cat_rank
    FROM product_revenue
)
SELECT category_name, product_name, total_revenue
FROM ranked
WHERE cat_rank = 1
ORDER BY total_revenue DESC;

Question 7: How did revenue trend month by month, with MoM percentage change?

Introduces LAG() for period comparisons — the core of time-series reporting.

    SELECT
        DATE_FORMAT(o.order_date, '%Y-%m')    AS month,
        COUNT(DISTINCT o.order_id)             AS orders_placed,
        SUM(oi.quantity * oi.unit_price)       AS revenue
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id

Question 8: Full monthly executive report — revenue, running total, and MoM change in one query.

The complete synthesis: CTE, JOIN, GROUP BY, and three window functions working together. This is the kind of query that populates an executive dashboard.

SQL · MySQL
WITH monthly AS (
    SELECT
        DATE_FORMAT(o.order_date, '%Y-%m')    AS month,
        COUNT(DISTINCT o.order_id)             AS orders_placed,
        SUM(oi.quantity * oi.unit_price)       AS revenue
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY DATE_FORMAT(o.order_date, '%Y-%m')
)
SELECT
    month,
    orders_placed,
    revenue,
    SUM(revenue) OVER (
        ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    )                                                            AS running_total,
    ROUND(
        (revenue - LAG(revenue) OVER (ORDER BY month))
        / LAG(revenue) OVER (ORDER BY month) * 100
    , 1)                                                         AS mom_pct_change
FROM monthly
ORDER BY month;

Quick Recap — Day 9

    Window functions compute across rows while keeping every row visible — unlike GROUP BY, which collapses rows.

    The OVER clause defines the window: PARTITION BY groups rows, ORDER BY sequences them, and the frame clause (ROWS BETWEEN ...) controls which rows are included in each computation.

    Ranking functions:

    • ROW_NUMBER() — unique sequential number, no ties.
    • RANK() — tied rows share a rank; next rank skips.
    • DENSE_RANK() — tied rows share a rank; next rank does not skip.

    **LAG(col, n) and LEAD(col, n)** return values from rows before and after the current row — essential for period-over-period comparisons.

    **SUM() OVER (ORDER BY ... ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) computes running totals. SUM() OVER ()** with no ORDER BY gives the grand total in every row — use it for percentage calculations.

    Views (CREATE VIEW name AS SELECT ...) save a query under a name, making it queryable like a table without storing data.

    Best practices: name columns explicitly, comment your logic, format for readability, use CTEs for complex queries, filter early, develop with LIMIT, and always sanity-check your totals.
